# `bridge/` — Rust core ↔ Flutter UI

This directory hosts the FFI surface between the Flutter UI in
`flutter_godesk/` and the RustDesk-derived Rust core in the parent
`client/` repo. Phase 2.4 deliverable.

## Contract

The Rust core exposes (broadly) these capabilities — Flutter consumes them
through this directory:

| Capability | Direction | Files |
|------------|-----------|-------|
| Local identity (`myId`, password regen, copy state) | Rust → Flutter | `identity.dart` |
| Address book (saved peers + last seen) | Rust ↔ Flutter | `peers.dart` |
| Connect / disconnect / session lifecycle | Flutter → Rust → Flutter (events) | `session.dart` |
| Stage events (resolving / tunnel / auth / connected) | Rust → Flutter | `session.dart` |
| Transfer queue (upload / download / progress / cancel) | Flutter → Rust → Flutter | `transfers.dart` |
| Permissions state (accessibility / screen / input on macOS, none on Windows) | OS → Rust → Flutter | `permissions.dart` |
| Diagnostics (relay name, encryption suite, latency, NAT type) | Rust → Flutter | `diagnostics.dart` |
| Settings (image quality / volume / mic gain / startup behaviors / relay region) | Flutter → Rust | `settings.dart` |

## Implementation status

🟡 **Skeletons only.** Each `.dart` file in this directory:

1. Defines the data class the UI consumes (mirrors `lib/data/*` mock types so
   migration is a near-search-replace at screens).
2. Exposes a `Bridge<X>` interface with `Stream` for events from Rust and
   `Future` for one-shot commands to Rust.
3. Ships a `MockBridge<X>` implementation that uses the existing mock data
   from `lib/data/`. The UI is wired to this by default.
4. Documents the Rust function it should call (matching what upstream
   `client/flutter/lib/` invokes through cargokit / flutter_rust_bridge).

When Phase 2.4 implementation lands, each file gets a `RealBridge<X>`
alongside the mock that calls the actual Rust crate via FFI. Switching at
`main.dart` is a one-line change.

## Why not direct calls already?

RustDesk's Flutter side uses a hand-written FFI through method channels +
`generated.bridge.dart` (auto-generated by their build pipeline). To call
the same Rust crate from our sibling package `flutter_godesk/`, we need to
either:

1. **Reuse the existing channel / generator** — wire our Flutter package to
   the same Rust crate path, run their codegen against our package, and
   import the generated bindings. Cleanest. Requires `client/build.py` patch
   to pass our package as the Flutter dir (per [ADR-011](../../wiki/decisions.md#adr-011--ui-overlay-strategy-sibling-flutter-package-flutter_godesk)).

2. **Stand up our own thin FFI** using `flutter_rust_bridge` v2. Maintains a
   clean separation but means we re-implement what upstream already wrote.
   Not worth it unless upstream's bindings cause pain.

Plan: try (1). If `build.py` accepts a `--flutter-dir` flag (single-line
patch), the rest is mechanical.

## What this isn't

This directory is **not** the place to put protocol definitions, encryption,
or networking — that all stays in the Rust core. We are the consumer, not
the implementor.
